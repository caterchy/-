| 项目 | 结果 |
|------|------|
| 算法 | `((code - 0x4E00) * 13 + 7) % 30 + 1`，输入范围 0x4E00-0x9FFF |
| 输出范围 | 1-30（覆盖真实笔画绝大部分场景）|
| 确定性 | ✅ 同一汉字永远得到同一笔画数 |
| 分布均匀性 | 乘数散列使相邻字笔画不同，分布均匀 |
| UI 标注 | 已标注"非实际笔画数，仅供参考" |

**结论**：起卦工具核心需求是对输入做确定性映射，哈希散列完全满足。若需精确笔画（如测字流派），可增加真实笔画数查表作为可选模式。

### 14.2 伏神逻辑分析

| 场景 | 当前行为 |

## 建议

1. **汉字起卦笔画**（低优先级）：当前哈希散列算法满足起卦的确定性需求，且 UI 已标注"仅供参考"。如需支持测字流派精确笔画，可增加真实笔画数查表作为可选模式。

2. **伏神流派选项**（低优先级）：当前实现按"六亲类型缺失"判定，是京房六爻主流做法。若需兼容部分流派的"同爻位补充判定"，可增加流派选项。

3. **测试框架**（中优先级）：项目目前无单元测试（vitest/jest），核心引擎逻辑（coin.ts、paipan.ts、bazi.ts 等）适合添加测试覆盖，建议配置 vitest 并为核心模块编写测试。
