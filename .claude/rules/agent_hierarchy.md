# Agent 层级和协作规则

## 层级结构

```
                     ┌─────────────────┐
                     │   总负责人 (我)   │  ← 决策中枢
                     └────────┬────────┘
           ┌──────────────────┼──────────────────┐
           ▼                  ▼                  ▼
   ┌───────────────┐  ┌──────────────┐  ┌──────────────┐
   │  私人助理      │  │  项目经理     │  │  技术执行组   │
   │  (规则维护)    │  │  (任务管理)   │  │              │
   └───────────────┘  └──────────────┘  │  ┌──────────┐ │
                                        │  │ 开发     │ │
                                        │  ├──────────┤ │
                                        │  │ Debug    │ │
                                        │  ├──────────┤ │
                                        │  │ 测试     │ │
                                        │  └──────────┘ │
                                        └──────────────┘
```

## 角色职责

| 层级 | 角色 | 职责 | 调用方式 |
|------|------|------|----------|
| L0 | **总负责人** | 需求分析、任务分配、结果审查、架构决策 | 默认角色，不通过 Agent 工具 |
| L1 | **私人助理** | 维护 `.claude/rules/` 规则文件，监督存储规范 | Agent 工具即时调用 |
| L1 | **项目经理** | 任务跟踪、进度管理、决策日志维护 | Agent 工具即时调用 |
| L2 | **开发工程师** | 实现新功能、代码重构 | Agent 工具即时调用 |
| L2 | **Debug 专家** | 根因分析、Bug 修复、性能优化 | Agent 工具即时调用 |
| L2 | **测试专家** | 编写测试、覆盖率分析、质量验证 | Agent 工具即时调用 |

## 协作规则

### 1. 总负责人调度规则
- 收到需求后先分类：开发 / Debug / 测试 / 管理 / 规则维护
- 对应角色无法完成时，由总负责人介入协调
- 跨角色问题（如 Debug 发现需要新功能）→ 退回总负责人重新分配

### 2. 私人助理规则
- **唯一拥有 `rules/` 目录写入权限的角色**
- 其他 Agent 对规则文件只有读取权限，不得修改
- 规则变更需记录到 `task-feedback/decision-log.md`
- 总负责人可要求私人助理创建或更新规则

### 3. 项目经理规则
- 任务方案必须先写 `task-plan.md`，经总负责人确认后再执行
- `checklist.md` 由项目经理维护，实时更新进度
- 架构/设计决策必须记入 `decision-log.md`

### 4. 技术执行组规则
- 开发 / Debug / 测试 严格按分配任务执行，不做范围蔓延
- 不自行修改规则文件或任务跟踪文件
- 完成工作后向总负责人报告，不自行动手验证

### 5. 冲突解决
- Agent 间意见不一致 → 升级到总负责人裁决
- 规则与实际情况冲突 → 私人助理评估后更新规则
